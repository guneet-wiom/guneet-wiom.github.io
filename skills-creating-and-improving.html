Here's the systematic loop. Write your skill instructions. Run it on at least 3 different inputs. Score the output against your expectations. Diagnose which dimension is weakest. Rewrite ONLY that part. Repeat. The discipline is: one fix per round. If you try to fix everything at once, you can't tell what helped. There's no fixed number of rounds — you stop when the output matches your standard. If someone asks how long a round takes: 5-10 minutes per round in a real workflow. The goal isn't speed — it's disciplined, incremental improvement. The next slide gives you a scoring rubric so you can make the Diagnose step objective, not a guess.

Here's your scoring rubric. Five axes, each scored 1 to 5. Accuracy: is the output factually correct and complete? Format: does it match the structure you asked for? Tone: does it sound right for the audience? Constraints: did it respect word limits, forbidden content, rules? Edge Cases: does it handle empty inputs, missing data, weird formats? You don't need a spreadsheet. Just eyeball which axis scored lowest. That's the one you fix next. This rubric is informed by prompt evaluation research (if anyone asks about PEEM or academic frameworks: it's an arXiv paper, good further reading but not needed today). This is a quick comparison — 90 seconds max. The key message: 'Write manually, test with skill-creator.' The manual loop teaches you the judgment you need. skill-creator automates the testing once you know what good looks like. Don't use skill-creator's Create mode as a substitute for thinking — it generates structure, not domain knowledge. The Eval and Improve modes are the real value: they run your skill with and without activation in parallel, score against assertions, and optimize your description field. Anthropic found activation problems in 5 of their own 6 official skills when they audited with skill-creator. If it helps Anthropic, it'll help you. But it costs tokens — so use it deliberately, not on every draft. For skills with side effects like deploying or sending messages, add disable-model-invocation: true in the frontmatter so Claude can't trigger them unexpectedly.

Design Patterns

Compose skills into systems

Overview

7 design patterns

Seven ways to compose skills into systems

1. Sequential Pipeline

Ordered steps, output feeds input

2. Fan-Out / Fan-In

Parallel subagents, merged results

3. Conditional Routing

Branching logic based on context

4. Multi-MCP Coordination

One skill, multiple tools

5. Iterative Refinement

Generate, validate, improve in a loop

6. Context-Aware Selection

Same goal, different tools per context

7. Domain Intelligence

Skill IS the expert, not just the connector

Problem-first: "I need to onboard a customer" → skill picks tools. Tool-first: "I have Slack MCP" → skill teaches best practices.
Seven patterns, each gets one slide. Spend 1-2 minutes per pattern — keep it moving. Before diving in, frame the two starting points from Anthropic's guide: Problem-first means you start with an outcome and the skill figures out which tools to use. Tool-first means you have a connected tool and the skill teaches Claude the best way to use it. Most skills lean one direction. Wiom examples: Pipeline = morning brief builder, Fan-Out = department reports, Conditional = ticket classifier, Multi-MCP = daily brief, Iterative Refinement = report generation with quality checks, Context-Aware = file storage routing, Domain Intelligence = campaign compliance checks.
Pattern 1

Sequential Pipeline

One agent, ordered steps

Fetch Slack threads
→
Extract action items
→
Format as checklist
→
Post to channel
Each step's output → next step's input

When to use

Clear linear dependencies. Each step must complete before next starts.

Key insight

Most common pattern. If you only learn one, learn this.

Sequential Pipeline is the most common pattern. Each step's output feeds into the next step's input. Like an assembly line. Real example: fetch Slack threads, extract action items, format as checklist, post to channel. If someone asks 'how does Claude know what order to run steps in?' — you literally number them in the instructions: '1. Fetch... 2. Extract... 3. Format...' Claude follows numbered instructions in sequence. Wiom example: the morning brief builder — fetch calendar, then fetch Slack, then format, then post.
Pattern 2

Fan-Out / Fan-In

Multiple agents, one goal

Orchestrator
→
Subagent: Sales Analysis
→
Subagent: Ops Analysis
→
Subagent: Mktg Analysis
→merge
Merge Report
Each branch = separate subagent with own context window doing complex reasoning

When to use

Independent tasks that each need their own thinking. No dependencies between branches.

Key distinction

Each branch is a separate subagent with its own context window. Expensive but powerful.

Fan-Out is when you spawn multiple SUBAGENTS in parallel. The key word is subagents — each branch gets its own context window (remember context windows from Session 1? Each subagent is like hiring a new person for that task — they start fresh, with no knowledge of what the other agents are doing). An orchestrator kicks off all three, they work in parallel, then results merge. Example: one subagent analyzes sales, another ops, another marketing. Each reasons deeply over different data. Expensive: full context window per branch. Use it when each branch truly needs independent thinking. Wiom example: department reports where each team's data needs separate analysis.
Pattern 3

Conditional Routing

One agent, branching logic

Analyze Input
→
Bug? →
Bug Triage Skill
Feature? →
Spec Writer Skill
Other? →
General Response
The router is itself a skill — its job is to decide, not to do

When to use

Different input types need completely different handling.

Key insight

The router skill evaluates context then delegates. Separation of concerns.

Conditional Routing is a branching pattern. A router skill inspects the input and decides which downstream skill to invoke. Example: an incoming message gets classified. Is it a bug report? Route to the bug triage skill. Feature request? Route to the spec writer skill. Something else? General response. The router is itself a skill, but its only job is to decide which path to take. It doesn't do the actual work. This is separation of concerns: one skill decides, another skill executes.

Multi-MCP Coordination

One agent, multiple tools

transform, error handling

→ messages

ONE skill initiates all calls, ONE context window — each call is a simple fetch, not complex reasoning

vs Fan-Out

Fan-Out = subagents + thinking
Multi-MCP = one skill + fetches

Rule of thumb

Needs thinking? → Fan-Out
Just a fetch? → Multi-MCP

Multi-MCP Coordination is one skill calling multiple external tools. Key distinction from Pattern 2 Fan-Out: this uses ONE context window. The skill is the glue logic that sequences calls, transforms data between them, and handles errors. Example: a daily brief skill calls Calendar MCP for today's meetings, Slack MCP for overnight messages, Sheets MCP for latest KPIs, then synthesizes everything into one formatted Slack message. Each call is a simple fetch, not complex reasoning. That's why you don't need separate subagents. Rule of thumb: if each branch needs independent thinking, use Fan-Out with subagents. If each branch is just a fetch or transform, use Multi-MCP in one skill. Fan-Out is expensive (full context window per branch). Multi-MCP is cheap (one context window total).
Pattern 5

Iterative Refinement

Generate, validate, improve — automatically

Generate draft
→
Run validation
→
Issues found?
No →
Finalize & save
Yes →
Fix issues & re-validate
↻ loop

When to use

Output quality improves with iteration. Reports, code generation, content creation.

Key insight

This bakes the iteration loop INTO the skill itself. The skill self-improves — not the human.

Iterative Refinement is the pattern from Anthropic's official guide. It directly connects Block 1 to Block 2: in Block 1 YOU did the iteration loop manually. This pattern bakes that same loop INTO the skill itself. Example: a report generation skill that drafts the report, runs a validation script to check for missing sections and formatting issues, fixes them, and re-validates until a quality threshold is met. Key techniques: explicit quality criteria (what counts as 'done'), validation scripts or checks, a maximum iteration count (so it doesn't loop forever), and finalization steps. Wiom example: a campaign brief skill that generates a draft, checks it against brand guidelines, fixes violations, and re-checks.
Pattern 6

Context-Aware Selection

Same goal, different tools depending on context

Check message
→
Urgent? →
Slack DM
Team update? →
Slack channel
External? →
Email via Gmail
The skill has a decision tree that picks the right tool for the job

vs Conditional Routing (Pattern 3)

Routing sends to different skills. Context-Aware selects different tools within one skill.

Key technique

Clear decision criteria + fallback options + transparency about why that tool was chosen.

Context-Aware Tool Selection is from Anthropic's official guide. It looks similar to Conditional Routing but the distinction matters: Routing sends to different SKILLS. Context-Aware selects different TOOLS within one skill. Example on screen: a 'send update' skill that checks the message type — urgent goes as a Slack DM, team updates go to a Slack channel, external communications go via Gmail. Key techniques: clear decision criteria (not vibes), fallback options if the preferred tool is unavailable, and transparency — the skill explains WHY it chose that channel.
