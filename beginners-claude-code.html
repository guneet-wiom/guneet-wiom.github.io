Jargon: Agent

What is an "agent"?

💼

Before defining Claude Code, make sure they understand the word "agent." Most people already know what an agent is in real life — a travel agent, a real estate agent, an insurance agent. What do they all have in common? They have specialized tools, domain knowledge, and they take action on your behalf. You tell the travel agent "I need to fly to Mumbai on Friday" — they don't just tell you about flights, they BOOK one. An AI agent is the same concept but in software. It has tools (file editing, running commands, searching), it has knowledge (the AI model), and it takes action autonomously. The key distinction from a chatbot: a chatbot answers questions, an agent does things. Claude Code is an AI agent.

Claude Code has a permission system — it CANNOT change anything without your approval. Reading files is auto-approved (safe). Editing files, running commands — requires your click. Claude Code has three modes that control how much freedom it has. Normal mode (default): Claude reads your files, makes a plan, and asks your permission before each change. This is where beginners should stay. Auto-Accept mode: Claude makes changes WITHOUT asking. Faster for experienced users who trust Claude, but risky for beginners — it will just edit files directly. Use with caution. Plan mode: Claude reads and analyzes but does NOT change any files. It proposes a plan and you review before anything happens. Best for complex or risky tasks where you want to understand the approach first. You can switch modes using /plan command or by clicking the mode indicator in the Desktop App input bar. Beginners: stay in Normal mode. Use Plan mode for big tasks. Avoid Auto-Accept until you're comfortable.

Jargon: Terminal

Two ways to use a computer

🖱 GUI

Point and click

⌨ Terminal

Type commands

Two ways to interact with a computer. GUI = Graphical User Interface — windows, icons, menus, mouse clicks. Everything they already know: PowerPoint, Chrome, File Explorer. CLI = Command Line Interface — you type a command, the computer responds in text. Like texting your computer instead of pointing and clicking. The jargon: Terminal, Command Prompt, PowerShell, CLI, Shell — all mean roughly the same thing. Different names across operating systems. Windows has Command Prompt, PowerShell, and Windows Terminal. Mac has Terminal. Don't let the names confuse you — same concept.

Four ways to access a terminal, all equally valid. (1) Windows Terminal: built into Windows, search "Terminal" or "Command Prompt" in Start. (2) VS Code Terminal: if you use VS Code (a code editor), there's a built-in terminal. Go to View menu, click Terminal. Many developers prefer this because code and terminal are side by side. Jargon: How AI Reads

Context Window & Tokens

Tokens

Word fragments. More = more cost.

Context Window

AI's working memory. Fills up.

Two terms they'll hear constantly. Tokens: the basic unit AI uses to process text. Not exactly words — AI breaks text into smaller chunks called tokens. Roughly 1 word = 1.3 tokens. Every token costs money (fractions of a cent). Every file Claude reads, every command it runs, every response it gives — all measured in tokens. Context window: think of it as Claude's short-term working memory. It has a fixed capacity (measured in tokens). Everything in the conversation takes up space: your prompts, Claude's responses, file contents it read, command outputs. When the context fills up, Claude starts losing track of earlier information. This is why /compact and starting fresh matter — they free up context space. Claude Code uses more context than regular chat because it reads entire files and runs commands.

Claude Code reads entire files, runs commands, accumulates a LOT of context. The "glass" fills up faster than regular chat. When full, Claude gets confused or forgetful. Three tools: /context shows a visual grid — each block represents a piece of the conversation, colored by type (your messages, Claude's responses, file reads, tool outputs). Three levels of CLAUDE.md. Global: your personal preferences applied everywhere — "always use English," "I prefer concise answers." This lives in your home directory. Project: project-specific rules committed to git and shared with your team — "use React," "components go in src/." Rules folder: modular topic-specific rules — security.md, testing.md, design.md. Claude loads all three levels automatically, every session. Rules at the bottom are the most specific and override rules above them when there's a conflict. Think of it as layering: your global preferences form the base, project rules add on top, and topic-specific rules in .claude/rules/ are the final layer.

Security: Secrets

Never put passwords in code.

.env 
# Database
DB_PASS=s3cureP@ss!

# API Keys
API_KEY=sk-live-xyz...
GH_TOKEN=ghp_abc...

🔒 This file stays on YOUR machine only.

✅ Right

password = process.env.DB_PASS
api_key = process.env.API_KEY
token = process.env.GH_TOKEN

Loaded from .env file.
Never committed. Never shared.

This is a critical habit to build from day one. API keys, passwords, tokens — these are "secrets." If you put them directly in your code and push to GitHub, anyone can see them. Companies have been hacked this way. The solution: store secrets in a .env file (environment file). Your code reads from .env instead of having the real values. The .env file stays on YOUR machine — never pushed to GitHub, never shared. Claude Code follows this rule too — it will NEVER hardcode secrets. If you paste a secret into chat, Claude will warn you and tell you to put it in .env instead.

How To

Creating a .env file

Step by step

1. Create a file named .env
   (yes, it starts with a dot)

2. Add your secrets, one per line:
   API_KEY=your-key-here
DB_PASSWORD=your-password
SLACK_TOKEN=xoxb-your-token

3. Your code reads from .env
   using a "dotenv" library

Rules

❌ Never commit .env
❌ Never share .env on Slack
❌ Never paste keys in code
❌ Never screenshot .env

✅ One .env per machine
✅ Share a .env.example
   (with fake values)
✅ Ask Claude to set it up

Ask Claude: "Set up a .env file for this project" — it knows the pattern.

Walk through creating a .env file. The dot prefix makes it a "hidden file" on Mac/Linux — it won't show in normal file browsers. Each line is KEY=VALUE, no spaces around the equals sign, no quotes needed (unless the value has spaces). The .gitignore file tells Git to ignore .env — so it never gets committed or pushed to GitHub. A .env.example file has the same keys but with placeholder values — this IS safe to commit because it doesn't contain real secrets. It shows the team what keys they need. Pro tip: just ask Claude Code to "set up a .env file for this project" and it will create both .env and .env.example with the right structure. Claude will also add .env to .gitignore automatically.

Security: .gitignore

Tell Git what to hide.

What is .gitignore?

A list of files Git
should never track or push.

One file. Lives in your project root.

Example .gitignore

# Secrets
.env

# Dependencies
node_modules/

# Build output
dist/
.next/

Ask Claude: "Add a .gitignore for this project" — it picks the right rules. App + Terminal

This pairs directly with the .env slide. A .gitignore file is a plain text list of files and folders that Git should NEVER track, commit, or push to GitHub. If .env is your vault, .gitignore is the lock that keeps the vault out of Git. Without .gitignore, you might accidentally push your .env file (with real passwords) to GitHub — where anyone can see it. Common entries: .env and .env.local (secrets), node_modules/ (downloaded dependencies — huge folder, no need to share), dist/ and .next/ (build outputs that can be regenerated). Claude Code adds .env to .gitignore automatically when it creates one. But always double-check: if .gitignore doesn't exist or doesn't include .env, ask Claude "Add a .gitignore for this project" — it knows the standard patterns for every tech stack.
