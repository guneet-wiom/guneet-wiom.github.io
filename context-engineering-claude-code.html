Why Context
Engineering?

The problem with "type and hope"

Show of hands: how many of you have typed something into Claude Code, gotten a response, and thought "that is not what I meant"? Almost everyone. Let us talk about why.
Sound familiar?

The "type and hope" problem

Claude forgets your preferences

You told it to use a certain format yesterday. Today it guesses again.

Long sessions go off the rails

Claude starts strong but after 30 minutes, it "forgets" what you asked earlier.

You keep repeating yourself

"Use the Wiom tone." "Pull from Metabase." "Format it like last time." Every. Single. Session.

It reads the wrong files

Claude reads 50 files to understand your project when you know it only needs 3.

These are not Claude being dumb. Claude is extremely capable. The problem is that Claude has a limited working memory called the context window, and if you do not manage it, you get bad results. Every one of these problems has a specific fix we will cover today.
Core concept

The context window

Think of it as Claude's working memory

🧠

Human working memory

You can hold about 7 things
in your head at once

🗒

Claude's context window

Holds everything:
conversation + files + instructions + tool definitions

The context window is like a glass of water. It holds your entire conversation, every file Claude reads, every command output, your CLAUDE.md instructions, tool definitions. And here is the key insight from Anthropic's own research: as it fills up, Claude gets worse, not better. This is called context rot. The transformer architecture means every token pays attention to every other token, so the more stuff in there, the more diluted Claude's attention becomes. This is why long sessions go off the rails.

Before we go further, we need to understand tokens. Tokens are not words. Claude breaks text into sub-word chunks. Simple English words are usually 1 token. Longer words get split. Numbers, symbols, and non-English characters cost more tokens. Why does this matter? Because the context window has a token LIMIT. Claude Sonnet has about 200,000 tokens. Opus can go up to 1 million. Everything you do — your messages, Claude's replies, file reads, tool definitions, CLAUDE.md — all eat from the same pool. When the pool fills up, that is when context rot begins.

This is the most important concept. Claude does not read like you. You read line by line, sequentially. Claude reads EVERYTHING in the context window simultaneously and compares every piece of information to every other piece. This comparison process is called "attention." Think of attention like a spotlight at a concert. If there is one performer on stage, the spotlight is bright and focused. If there are 500 performers on the same stage, that same spotlight has to cover everyone — and each person gets much less light. That is exactly what happens when your context window fills up.

This is the key visual. Show the graph. Claude starts near 100% accuracy with a small context. As the window fills, accuracy degrades along a curve — slowly at first, then accelerating. The sweet spot is roughly under 50K tokens. Beyond that, you are in diminishing returns territory. At a full 200K window, Claude is noticeably less accurate than at 50K. Two critical effects to mention: First, "lost in the middle" — research shows Claude pays more attention to information at the start and end of the context window. Information buried in the middle gets overlooked. This is why CLAUDE.md loads at the top and why repeating instructions mid-conversation is less effective. Second, context rot is a GRADIENT, not a cliff. Claude does not suddenly stop working. It gradually gets worse. You might not even notice at first — it just starts missing details, forgetting instructions, making small mistakes. By the time you notice, the window is already polluted. The solution? Everything we are about to learn: CLAUDE.md for persistent top-of-context rules, Skills for on-demand loading, /clear between tasks, /compact when sessions get long.
The shift
This diagram is from Anthropic official engineering blog. On the left: prompt engineering for single turn queries. Simple: system prompt plus user message goes in, assistant message comes out. On the right: context engineering for agents. You have a pool of possible context: docs, tools, memory files, comprehensive instructions, domain knowledge, message history. The curation step selects what goes into the context window. Not everything, just the right things. System prompt, relevant docs, memory, relevant tools, user message, message history. Then the model produces not just a response but also tool calls, which add more to context. And the scissors at the bottom: when context gets too big, you cut. That is compaction. The key insight: context engineering is iterative. Curation happens every time we decide what to pass to the model.

What is Markdown? How do file paths work?

Foundation 1

What is Markdown?

The language that powers AI configuration everywhere

What you type (.md file)

# My Rules

## Code Style
- Use **Wiom brand voice**
- Format numbers in lakhs

## Workflow
Verify data with `Metabase` before sharing

Why it matters for AI

Human-readable — no special software

Structured — headings, lists, code blocks

Universal — CLAUDE.md, SKILL.md, MEMORY.md, GitHub, Notion, Slack

Lightweight — just a text file with formatting

Key syntax: # Heading   **bold**   - bullet   `code`
Markdown is the file format that powers all of Claude Code configuration. A .md file is just a text file with simple formatting. Hash creates headings. Double asterisk makes bold. Dash creates bullets. Backticks create code. No special editor needed, Notepad works. Why does AI use markdown? Perfect middle ground: structured enough for machines, readable for humans. Every file we work with today ends in .md. If you can write a WhatsApp message with bullet points, you can write markdown.
Foundation 2

File paths: where things live

Where you put a file determines who it affects

Home directory ~/

Your personal space. Applies to all your projects. Only you see it.

Windows: C:\Users\yourname\
Mac/Linux: /Users/yourname/

Project directory ./

The folder you are working in now. Applies to this project only. Shared with team via git.

Example: C:\Users\you\projects\wiom-dashboard\

Claude config ~/.claude/

Hidden folder in your home directory. Claude Code stores config, memory, and your personal skills here.

Project config .claude/

Hidden folder in your project for shared skills, rules, settings. Committed to git = shared with team.

File paths are addresses for files on your computer. Home directory is your personal space, files affect all your projects but only for you. Project directory is the folder you are working in, shared with teammates via git. Two hidden folders: ~/.claude/ for YOUR personal config, memory, and personal skills. .claude/ in a project for shared team skills, rules, settings. Key takeaway: home = personal, project = team-shared.

CLAUDE.md

Make Claude follow your rules — every session, automatically

This is the single most impactful thing you can set up. CLAUDE.md is a file that Claude reads at the start of EVERY conversation. Your persistent rulebook.
The basics

What is CLAUDE.md?

A markdown file Claude reads first — every single session

You write rules, preferences, and project context in plain text.

Claude loads it automatically at the start of every conversation.

It survives /compact — Claude re-reads it fresh from disk.

Instructions given only in conversation get lost. CLAUDE.md is permanent.

Quick start

$ claude
> /init
Analyzing your project...
Created CLAUDE.md

Type /init to auto-generate one

CLAUDE.md is a plain markdown file Claude reads at the START of every session. Your persistent rulebook. Critical insight: CLAUDE.md survives /compact. When you hit context limits and Claude compresses your conversation, it re-reads CLAUDE.md from disk fresh. But instructions given only in conversation may get lost. That is why anything important should be in CLAUDE.md. Fastest way to start: type /init. Claude analyzes your project and generates a starter CLAUDE.md.
Scope

Two places, two purposes

Global — just for you

~/.claude/CLAUDE.md

Your personal preferences across ALL projects

"I prefer concise responses"
"Format numbers in Indian notation (lakhs, crores)"
"I am in marketing, not engineering"

Project — for your team

./CLAUDE.md

Shared standards for this project

"Reports use Metabase-sourced data"
"Always format currency in rupees with commas"
"Marketing templates live in the Canva team folder"

Priority: When rules conflict, project CLAUDE.md wins over global. More specific = higher priority.
Global at ~/.claude/CLAUDE.md applies to every project. Put personal preferences here. Project at ./CLAUDE.md applies to this project only, shared with team via git. When there is a conflict, project rule wins. Commit your project CLAUDE.md to git so the whole team gets the same Claude behavior.
What goes in?

What to put in CLAUDE.md

Include

Build commands Claude cannot guess
Code style rules that differ from defaults
Test instructions and preferred runners
Project architecture decisions
Common gotchas and quirks
Your role and communication style

Do not include

Things Claude already does correctly
Standard language conventions
Detailed API docs (link instead)
Info that changes frequently
Self-evident practices
Task-specific workflows (use Skills)
The test: For each line, ask "Would removing this cause Claude to make mistakes?" If not, delete it.
Straight from Anthropic best practices. Include: things Claude cannot figure out on its own. Exclude: things Claude already knows, things that change often, detailed docs (link with @path instead). The test: would removing this cause mistakes? If no, delete it. Bloated CLAUDE.md where rules get lost is worse than none.
Examples

Specific beats vague

Vague (Claude ignores)

# Guidelines
- Format things properly
- Check your work
- Keep things organized
- Write clearly
- Follow standards

Specific (Claude follows)

# Data & Reports
- Format currency in lakhs
- Use Metabase as single source of truth

# Quality
- Verify data before sharing externally
- Keep dashboards under 10 widgets

# Standards
- Marketing assets in the Canva team folder
- Use active voice in client-facing docs
Most common mistake. Vague instructions like "format things properly" waste context. Specific instructions like "format currency in lakhs" are verifiable. If you cannot verify whether Claude followed it, it is too vague.

Auto Memory

Claude takes notes so you don't have to

CLAUDE.md is what YOU write. Auto Memory is what CLAUDE writes. Complementary systems.
Two systems

You write rules. Claude takes notes.

CLAUDE.md — you write

Instructions and rules
Project, user, or org scope
Full file loaded every session
Coding standards, workflows, architecture

Auto Memory — Claude writes

Learnings and patterns
Per project (machine-local)
First 200 lines loaded per session
Build commands, debugging insights, your preferences

How it works: Claude stores notes at ~/.claude/projects/your-project/memory/. A MEMORY.md index (first 200 lines) loads every session. Detailed notes in separate topic files load on demand.
Two complementary systems. CLAUDE.md: YOU write the rules. Auto Memory: CLAUDE writes notes for itself. When you correct Claude, it saves that. When it discovers something through debugging, it saves that. Storage: each project gets a memory directory. MEMORY.md index loads first 200 lines every session. Topic files load on demand. Auto Memory is machine-local, not shared with teammates.

"What do you remember about this project?"

Browse and edit

Type /memory to see loaded files, toggle auto memory, open files in your editor.

Pro tip: Memory files are just markdown. Edit or delete them directly. If Claude remembers something wrong, fix the file.
Three ways to interact. Tell Claude to remember something explicitly. Ask what it knows. Use /memory to browse everything. Memory files are just markdown, edit or delete them directly. You are always in control.

SKILL.md stays focused — overview only

Supporting files lazy-load — Claude reads only when needed

Scripts can execute — Claude runs helper scripts

Keep SKILL.md under 500 lines

A skill is a directory, not just a file. SKILL.md is the entry point. Add templates, examples, scripts. Supporting files are lazy-loaded, they only enter context when Claude needs them. You can bundle a 2000-line reference as a supporting file, it only loads when relevant. Scripts can execute: data gathering, validation, any helper you bundle.

Rule of thumb: If the skill has side effects (deploys, sends messages, deletes), set disable-model-invocation: true. You do not want Claude deciding to publish a report without your review.

By default, both you and Claude can trigger any skill. Set disable-model-invocation for skills with side effects: deployments, sending messages, deleting data. The system

Why the 200-line rule matters

CLAUDE.md

200

Soft limit. Full file loads every session.

Longer = lower adherence. Claude starts ignoring rules buried in a long file.

Every token loads with every single message.

MEMORY.md

200

Hard limit. Only first 200 lines load at session start.

Anything beyond line 200 is not loaded.

Claude keeps it concise by moving details into topic files.

Skills

∞

No limit. Only a one-line description loads at startup.

Full content loads only when invoked.

The overflow valve for everything that does not fit in CLAUDE.md.

The principle: CLAUDE.md for rules you need every session. Memory for things Claude learns. Skills for everything else. This is how you keep your context lean.
This is the connecting slide. Three systems, three loading behaviors. CLAUDE.md: soft 200-line recommendation. The full file loads every session regardless of length, but longer files mean Claude pays less attention to each rule. Think of it like a to-do list: 5 items and you do them all. 50 items and you start skipping. MEMORY.md: hard 200-line limit. Only the first 200 lines load at session start. Content beyond line 200 is simply not loaded. Claude handles this by keeping MEMORY.md as a concise index and moving detailed notes into separate topic files that it reads on demand. Skills: no limit at all. Only the description (one line) loads at startup so Claude knows the skill exists. The full skill content only loads when you type the slash command or Claude decides to invoke it. This is why Skills are the answer when your CLAUDE.md grows too long. Move specialized workflows from CLAUDE.md into Skills. Your context stays lean, but the knowledge is still available on demand.

Six commands. /context: diagnostic, shows what consumes your window. /clear: nuclear reset between unrelated tasks, the single most impactful habit. /compact: softer, summarizes so you can keep going. /rewind: restore conversation and code to a checkpoint. Decision framework

Clear, compact, or new session?

/clear

When: Switching to unrelated task

Effect: Full reset

Rule: Use between every unrelated task. It is free.
